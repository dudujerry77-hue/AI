# Planner Engine

Milestone 1 foundation package for the Titan Core Planner Engine.

Milestone 2 adds Planner domain model definitions and typed API signatures only.

Milestone 3 adds deterministic goal analysis and goal decomposition, and implements `createPlan()`.

Milestone 4 adds deterministic structural plan validation, and implements `validatePlan()`.

## Scope (Milestone 1)

- Implements the Titan runtime engine contract via lifecycle, health, metadata, version, and state methods.
- Exposes Planner API method signatures:
	- `createPlan()`
	- `validatePlan()`
	- `optimizePlan()`
	- `estimatePlan()`
	- `explainPlan()`
	- `cancelPlan()`

## Planner Domain Model (Milestone 2)

The planner package defines immutable domain types for:

- goals: `Goal`, `GoalType`, `GoalPriority`, `GoalStatus`
- plans: `Plan`, `PlanStatus`, `PlanMetadata`
- steps: `PlanStep`, `StepType`, `StepStatus`
- tasks: `Task`, `TaskStatus`
- dependencies: `Dependency`, `DependencyType`
- constraints: `Constraint`, `ConstraintType`
- planning context: `PlanningContext`
- estimates: `PlanEstimate`, `CostEstimate`, `TimeEstimate`, `ResourceEstimate`
- explanation: `PlanExplanation`

These models are exported through the planner package entry point and used by PlannerEngine API method signatures.

## Goal Analysis & Decomposition (Milestone 3)

Milestone 3 implements the first real Planner behavior, scoped strictly to deterministic,
offline, structural processing of a single `Goal`:

### `GoalAnalyzer`

- Analyzes a single `Goal` object only.
- Validates required fields (`goalId`, `title`, `description`, `type`, `priority`, `status`,
  `createdAt`, `updatedAt`).
- Validates enum values (`GoalType`, `GoalPriority`, `GoalStatus`).
- Validates timestamps (`createdAt`/`updatedAt` must be valid ISO-8601 timestamps, and
  `updatedAt` must not precede `createdAt`).
- Returns a structured `GoalAnalysisResult` (`{ goalId, valid, issues }`).
- Does **not** access the Context Engine.
- Does **not** access the Knowledge Engine.
- Does **not** use AI.
- Does **not** perform optimization.

### `GoalDecomposer`

- Deterministically converts one `Goal` into one `Plan`.
- Produces a fixed static sequence of `PlanStep`s: Analysis → Design → Implementation →
  Validation → Documentation.
- Produces one `Task` per `PlanStep`.
- Produces sequential `Dependency` edges between consecutive steps, plus `requires` edges
  linking each task to its step.
- No scheduling, no execution, no optimization, no graph algorithms, no learning, no
  orchestration.

### `PlannerEngine.createPlan()`

- Analyzes the request's `Goal` using `GoalAnalyzer`.
- If the goal is invalid, throws `PlanningValidationError` (with the structured issue list).
- Otherwise, returns the `Plan` generated by `GoalDecomposer`.

### Shared Planner Errors

`engines/planner/src/errors/planner-errors.ts` hosts:

- `NotImplementedError` — thrown by planner APIs that remain unimplemented.
- `PlanningValidationError` — thrown by `createPlan()` when goal analysis fails (carries the
  structured `GoalValidationIssue[]` list), and thrown by `PlanValidator.validate()` only for
  malformed input.

## Plan Validation (Milestone 4)

Milestone 4 implements deterministic, offline **structural** validation of a `Plan`:

### `PlanValidator`

- Validates a single `Plan` object structurally:
  - Required fields (`planId`, `goalId`, `status`, `metadata`, `steps`, `tasks`,
    `dependencies`, `constraints`).
  - Enum values (`PlanStatus`, `StepType`, `StepStatus`, `TaskStatus`, `DependencyType`,
    `ConstraintType`).
  - `PlanMetadata` (`createdAt`/`updatedAt` ISO-8601 timestamps and ordering, `createdBy`,
    `revision >= 1`).
  - Each `PlanStep` (required fields, valid `type`/`status`, `taskIds` array).
  - Each `Task` (required fields, valid `status`, and that `stepId` references an existing
    step).
  - Each `Dependency` (required fields, valid `type`, and that `sourceId`/`targetId` reference
    an existing step or task).
- Detects duplicate `stepId`s, duplicate `taskId`s, and duplicate `dependencyId`s.
- Detects duplicate dependency edges (same `type` + `sourceId` + `targetId`).
- Rejects self-dependencies (`sourceId === targetId`).
- Returns a structured `PlanValidationResult` (`{ planId, valid, issues }`); structural
  problems are reported as issues, not thrown.
- Throws `PlanningValidationError` **only** for malformed input: `null`, `undefined`, or a
  non-object value.
- Does **not** perform graph traversal.
- Does **not** perform cycle detection.
- Does **not** optimize.
- Does **not** schedule.
- Does **not** execute.
- Does **not** call any other engine.

### `PlannerEngine.validatePlan()`

- Delegates entirely to `PlanValidator.validate()` and returns its `PlanValidationResult`.
- Does not modify `createPlan()`, `GoalAnalyzer`, or `GoalDecomposer`.

## Current Behavior

- `createPlan()` is implemented (Milestone 3): deterministic goal analysis + decomposition.
- `validatePlan()` is implemented (Milestone 4): deterministic structural plan validation.
- `optimizePlan()`, `estimatePlan()`, `explainPlan()`, and `cancelPlan()` remain stubs and
  throw `NotImplementedError`. No functionality for these four APIs is implemented in this
  milestone.
